Speckle development guidelines

How we build speckle: a modular monolith in idiomatic Go, driven by outside-in acceptance tests and red / green / refactor.

Rules of the road

  1. Acceptance test first, always. No production code ships without a failing acceptance test that drove it.
  2. Red → green → refactor. One cycle at a time. Don't refactor on red. Don't add scope on green.
  3. New behaviour lives in internal/<module>/. Small exported interface, unexported implementation, constructed in main.go.
  4. Modules don't reach into each other. Cross-module calls go through the interface. No imports of another module's private types.
  5. Wrap errors with %w; propagate context.Context through anything that can block.
  6. A bug starts as a failing reproducer. Reproduce in a test, then fix.
  7. If you genuinely can't acceptance-test something, say so in the commit. One-line note is enough; don't pretend.

1.Modular monolith

One binary, many modules. Each module is a single Go package under internal/ with an exported interface describing what it does and an unexported type implementing it. The binary's main.go is the only place that knows about concrete implementations.

Layout

speckle/
├── main.go                  # composition root only
├── internal/
│   ├── spec/                # parse + validate .speckle files
│   │   ├── spec.go          # exported interface + types
│   │   └── parser.go        # unexported implementation
│   ├── overlay/             # yaml.Node deep-merge
│   ├── server/              # HTTP handlers, SSE, fsnotify
│   └── render/              # HTML template + Markdown
├── examples/
├── docs/
└── go.mod

The shape of a module

Each internal/<module> exposes one or more small interfaces and a constructor. The constructor is the only exported function; everything else is the interface contract.

// internal/spec/spec.go
package spec

type Parser interface {
    Parse(b []byte) (*Spec, error)
}

type Spec struct { /* exported value types are fine */ }

func NewParser() Parser { return &parser{} }

// internal/spec/parser.go
package spec

type parser struct{}

func (p *parser) Parse(b []byte) (*Spec, error) { /* ... */ }

The implementation type is unexported. Callers can't type-assert their way into the private fields, and they can't construct one without going through NewParser.

Cross-module talk

If server needs to parse a spec, it accepts a spec.Parser in its constructor. It never imports the private parser type.

Do

// internal/server/server.go
func New(parser spec.Parser, ...) Server {
    return &server{parser: parser}
}

// main.go
parser := spec.NewParser()
srv    := server.New(parser, /* ... */)

Don't

// internal/server/server.go
import "github.com/ptetau/speckle/internal/spec"

func (s *server) handleIndex(...) {
    p := &spec.parser{} // unexported — won't compile
    p.Parse(b)            // and also: don't.
}

Where interfaces are defined

Define an interface in the package that uses it whenever the behaviour is being requested by a consumer. Define it in the package that provides it when that behaviour is the package's central abstraction. Most of the time the provider package is the right home — be consistent within a module.

No package-level state

No var globalThing = ... at package level (except //go:embed assets and similar truly-constant values). Everything passes through a constructor and is owned by a struct. This is what makes tests possible: nothing is "already wired" the moment you import.

Why this works in Go: Go's package boundary is strong (unexported identifiers can't leak), interfaces are structural (no inheritance ceremony), and zero values are useful (constructors stay tiny). The "modular monolith" doesn't need a framework — it needs you to respect internal/.

2.Outside-in TDD

Every change starts at the outside — a failing test against the user-observable behaviour we're about to add or fix. We drill inward with unit tests only as needed to make that acceptance test pass.

The cycle

  1. RedWrite the failing acceptance test. Exercise the capability through the CLI or HTTP entry point. Run it; confirm it fails for the right reason (not because the binary won't build).
  2. GreenDo the minimum to make it pass. If a piece of pure logic blocks the path, drop down: write a failing unit test for that logic, make it pass, climb back up. Repeat until the acceptance test passes.
  3. RefactorClean up with the bar still green. Rename, extract, deduplicate. Tests stay green throughout. Don't introduce new behaviour during the refactor step — that's the next cycle's red.

What "acceptance test" means here

Plain Go _test.go files, run by go test ./.... No browser, no Gherkin framework. Two flavours, chosen by what's cheapest to exercise the behaviour:

  • HTTP acceptance: spin up the real http.Handler from internal/server inside httptest.NewServer; drive it with a regular HTTP client. The handler runs with real parsing, real fsnotify-less state, real submission channel — no mocks of our own code.
  • CLI acceptance: either call the run<Cmd> function directly with synthesised args, or os/exec the built binary in a t.TempDir(). Use the binary when stdin/stdout buffering or process exit semantics are part of the contract; otherwise the in-process call is faster.

Acceptance tests live in the package that owns the entry point — HTTP acceptance tests beside internal/server, CLI acceptance tests beside main.go (or a sibling main_test.go). Prefer external-package (package server_test) so the test exercises the public surface, not the internals.

A worked acceptance test

// internal/server/submit_acceptance_test.go
package server_test

import (
    "encoding/json"
    "net/http"
    "net/http/httptest"
    "strings"
    "testing"

    "github.com/ptetau/speckle/internal/server"
    "github.com/ptetau/speckle/internal/spec"
)

func TestSubmitFlowsThroughToAwait(t *testing.T) {
    h := server.New(spec.NewParser(), fixtureSpecPath(t))
    ts := httptest.NewServer(h)
    t.Cleanup(ts.Close)

    // user submits
    body := strings.NewReader(`{"spec_version":1,"decisions":{"x":{"selected":"a"}}}`)
    res, err := http.Post(ts.URL+"/submit", "application/json", body)
    if err != nil || res.StatusCode != 204 {
        t.Fatalf("submit: %v %v", err, res.Status)
    }

    // agent awaits, gets that exact submission back
    res, err = http.Get(ts.URL + "/await")
    if err != nil { t.Fatal(err) }
    var got struct{ Decisions map[string]struct{ Selected string } }
    if err := json.NewDecoder(res.Body).Decode(&got); err != nil { t.Fatal(err) }
    if got.Decisions["x"].Selected != "a" {
        t.Fatalf("await returned %+v", got)
    }
}

Unit tests as scaffolding

Unit tests exist to help an acceptance test pass. They isolate one piece of pure logic — the overlay merge, the spec validator, a tricky parse case — so you can red / green fast without paying for HTTP setup each cycle. When you've climbed back up and the acceptance test passes, the unit tests stay; they document the contract of that internal function.

If a unit test has no acceptance test depending on it, ask why the code exists. Either the acceptance gap is the bug, or the unit test is exercising unreachable behaviour.

Bug fixes

Same loop. The bug report becomes the failing acceptance test: reproduce the failure mode through the public entry point, watch it fail, fix until green, refactor if needed. A bug-fix commit without a regression test is incomplete.

3.Go idioms

Errors

  • Return errors, don't panic. Panics are for programmer error (impossible-state assertions), not for malformed input or failed I/O.
  • Wrap with fmt.Errorf("doing X: %w", err). Always add context, always preserve the cause.
  • Define sentinel errors (var ErrNotFound = errors.New(...)) only when callers need to distinguish a category. Otherwise: just return the wrapped error.
  • Never log and return the same error. Pick one. Returning is almost always right.

Context

  • context.Context is the first parameter of any function that does I/O, waits, or starts a goroutine.
  • Don't store ctx in a struct. Pass it down per call.
  • context.Background() appears only at the composition root (main.go) and in tests.

Concurrency

  • Every go statement comes with a plan for when the goroutine exits. ctx.Done() is observed; long-running loops bail on shutdown.
  • sync.RWMutex for shared state with many readers and few writers (e.g., the current parsed spec). chan for hand-offs between producers and consumers (e.g., submission delivery to await).
  • Don't share state via package-level globals. Pass it.

Logging

  • log/slog with a JSON handler in production, text handler in dev. One logger constructed in main.go, passed to modules that need it.
  • Log at warn/error for unexpected conditions; info for lifecycle events (server start, file reload); never info in a hot path.

Interfaces

  • Small. One to three methods. Six is two interfaces.
  • Named by capability (Parser, Watcher, Submitter), not by implementation (YAMLParser).
  • Accept interfaces, return concrete types — except at module boundaries, where the constructor returns the interface to keep the implementation private.

Naming

  • Package names: short, lowercase, no underscores. spec, not specparser.
  • File names: lowercase.go or lower_with_underscores.go.
  • Exported identifiers: MixedCaps; unexported: mixedCaps. No stuttering: spec.Parser, not spec.SpecParser.

Tests

  • _test.go next to the source.
  • Prefer external-package tests (package foo_test) for anything that should exercise the public surface; same-package only when you genuinely need access to unexported helpers.
  • One behaviour per test. Table-driven for variations of the same behaviour. t.Run for sub-cases so failures point at the specific row.
  • t.TempDir(), t.Cleanup, t.Parallel() where safe.

4.A change, end-to-end

Worked example: adding a speckle validate <file> subcommand that checks a .speckle file without serving it. Exit code is the contract.

  1. Red Write the failing acceptance test. Build the binary, run speckle validate against a good file and a bad file, assert exit codes. 
    func TestValidate(t *testing.T) {
    bin := buildBinary(t)                       // helper, builds once
    good := writeTempSpec(t, validYAML)
    bad  := writeTempSpec(t, invalidYAML)

    if code := runExit(bin, "validate", good); code != 0 {
        t.Fatalf("good file: want 0, got %d", code)
    }
    if code := runExit(bin, "validate", bad); code != 1 {
        t.Fatalf("bad file:  want 1, got %d", code)
    }
}
    Run it. It fails because validate isn't a known subcommand. That's the right failure.
  2. Green Wire validate in main.go, call into spec.Parser, return an error on failure. If spec.Parser doesn't yet expose the precise check you need, drop down: write a failing unit test in internal/spec first, make it pass, climb back up. Stop when the acceptance test goes green.
  3. Refactor Now that everything is green, look at what you wrote. Is the wiring code duplicated with serve or patch? Could the parser error messages be clearer? Tidy. Tests stay green throughout. If you find yourself wanting to add a feature, stop: that's the next red.
  4. Commit. Message says what the user-visible change is, not what files moved. CI runs go test ./...; that's the gate.
Cadence: small commits over big ones. One red / green / refactor cycle per commit is a good default; squash later if you want a tidier history.

5.When to bend the rules

The rules exist because they make the codebase boring in a good way. Bend them when you have a clearer reason than convenience, and say so out loud.

  • Pure visual / CSS work. Typography, colours, spacing — automated tests rarely buy anything here. Ship behind manual review; commit message says "no acceptance test — pure visual change."
  • Experimental spike. Sometimes you genuinely don't know the shape yet. Spike on a branch named spike/<topic>, no test discipline, fully disposable. Before merging anything from a spike, rewrite it test-first.
  • Mechanical refactor. Renames, package moves, gofmt-only changes. No new acceptance test required, but the existing test suite must pass before and after the same way — that's the safety net.
  • Vendored / generated code. Not ours, not tested by us, not idiomatised. Treat as a black box at the boundary.

Anything not on this list: write the test first.
